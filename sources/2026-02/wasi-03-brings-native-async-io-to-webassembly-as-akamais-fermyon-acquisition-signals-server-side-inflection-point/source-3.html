WASI Standards Evolution Whitepaper: Feature Panorama from 0.2 to 0.3

A comprehensive standards evolution document, breaking down the core meta-model and Worlds defined in WASI 0.2, and deeply analyzing the new Future/Stream types, Async syntax, and IO interface refactoring in WASI 0.3.

By WasmRuntime Team•January 12, 2026•7 min read

The evolution of the WebAssembly System Interface (WASI) is reshaping how cloud-native components are developed. This article aims to provide a high-granularity feature evolution guide, helping developers and architects understand the specific technical transformations from WASI 0.2 (Preview 2) to WASI 0.3 (Preview 3).

1. Introduction: From "Modularization" to "Concurrency"

The evolution of WASI follows a clear logical thread:

  • WASI 0.2 (2024): Solved the "How to write components" problem. Through the Component Model and WIT (Wasm Interface Type), it standardized cross-language type definitions and module composition, laying the foundation for static structure.
  • WASI 0.3 (2025): Solves the "How to run efficiently" problem. By introducing Native Async and Stream Types, it addresses performance bottlenecks and programming pain points in IO-intensive scenarios present in version 0.2, completing the puzzle of the dynamic runtime.

2. Baseline: What did WASI 0.2 (Preview 2) Define?

To understand the improvements in 0.3, we must first clarify the baseline specifications established by version 0.2. WASI 0.2 is the stable standard currently fully supported by mainstream runtimes (like Wasmtime 17+).

2.1 Core Meta Model

The core contribution of WASI 0.2 lies in standardizing the system interface description language.

  • WIT (Wasm Interface Type): WASI 0.2 officially established .wit files as the carrier for interface contracts. It defines rich data types:

    • Basic types: u8, s32, float64, bool, string, char
    • Composite types: record (struct), variant (enum/union), list, tuple, flags, option, result

  • Worlds: 0.2 introduced the world keyword to define the complete environment context (i.e., the collection of Imports and Exports) required for a component to run.

2.2 Resource Handles

In WASI 0.2, all system resources (Files, Sockets, Timers) are abstracted as Resources.

  • Opaque Handles: Resources are opaque types in WIT, represented physically as i32 indexes.
  • Ownership Semantics:
    • borrow<resource-name>: Borrowing; the caller retains ownership.
    • own<resource-name>: Transfer; ownership is moved to the callee (usually implying destruction).

2.3 Standard Library Worlds Overview

WASI 0.2 standardized two core Worlds:

A. wasi:cli (Command Line Environment)

This is a modern mapping of POSIX, containing the following core packages:

  • wasi:filesystem:
    • descriptor (Resource): Corresponds to file descriptors, supports read, write, sync, stat.
    • preopens: Defines directory permissions mounted when the component starts.
  • wasi:environment: Access to argv and environment variables.
  • wasi:exit: Control process exit codes.
  • wasi:io (0.2 Core):
    • pollable (Resource): The core of the 0.2 async model. Any operation needing waiting (like reading a stream, delays) returns a pollable handle.
    • input-stream / output-stream (Resource): Abstractions for byte stream operations.

B. wasi:http (HTTP Proxy Environment)

Designed for Serverless and Gateways:

  • wasi:http/proxy: Defines incoming-handler for handling HTTP requests.
  • wasi:http/outgoing-handler: For initiating HTTP requests.
  • Resource Overload: In 0.2, to simulate asynchrony, HTTP defined many complex resources like future-incoming-response, which developers had to manually poll.

3. Core Increments: WASI 0.3 (Preview 3) New Features Detail

WASI 0.3 is not a complete rewrite but a precise surgical upgrade targeting the pain points of 0.2 (especially IO and Concurrency). Here are the new concepts introduced in 0.3.

3.1 New Additions to WIT Type System

WASI 0.3 injects concurrency primitives into the WIT type system, making them First-class Types.

A. future<T> (New)

In 0.2, a Future was a design pattern (usually a Resource); in 0.3, it is a native type.

  • Definition: Represents a computation that will produce a value of type T at some point in the future.
  • Features:
    • Generic Support: Can be future<string>, future<u32>, etc.
    • Ownership Transfer: future itself is Linear, meaning the value can only be consumed once.
    • Automatic Lifecycle: No need to manually call destructors like Resources; the Runtime manages it automatically.

B. stream<T> (New)

In 0.2, Streams could only handle u8 (byte streams); in 0.3, they are generic structured streams.

  • Definition: Represents an asynchronous sequence of values of type T.
  • Core Feature: Backpressure:
    • This is the most significant improvement in 0.3. The stream type has built-in backpressure mechanisms.
    • Push-Pull Model: When the consumer cannot keep up, the Runtime detects the buffer is full. The producer's write operation is then automatically Suspended until the consumer reads data and frees up space. This requires no complex flow control code in the application layer.

C. error-context (New)

  • Definition: A standardized error handle type.
  • Purpose: Allows passing rich error debug information (Traceback, Error Message) across component boundaries, solving the problem in 0.2 where error information was lost or only error codes could be passed.

3.2 New Syntax and Definitions

A. async Keyword

WIT syntax now allows directly marking functions as asynchronous.

  • Syntax:
    language-wit

    6
    // WASI 0.3 Style
interface key-value {
    // 'async' here means the function may suspend and return a future
    get: async func(key: string) -> result<string, error>;
}
  • Underlying Mapping (Task Concept):
    • At the Canonical ABI layer, async func is mapped to asynchronous operations scheduled via the Task mechanism.
    • Task is the fundamental unit of Structured Concurrency introduced in the Component Model. An async function call creates a new Task, and the Runtime is responsible for scheduling and switching (Yield/Resume) between Tasks.

4. Interface Refactoring: Detailed Diff of 0.2 vs 0.3

This section shows the specific Breaking Changes in the WASI Standard Library APIs.

4.1 Refactoring of the IO Model (wasi:io)

This is the part with the biggest changes, completely abandoning pollable.

Async Corepollable Resourcefuture<T> and async keyword
Wait Mechanismpoll-oneoff(list<pollable>)Native language await (Task Suspend underneath)
Input Streaminput-stream (Resource)stream<u8> (Native Type)

Interpretation of Changes:

  • Removal of pollable: Developers no longer need to maintain a massive pollable list for select operations.
  • Generic Streams: Besides byte streams, 0.3 allows defining structured streams like stream<log-entry>.

4.3 Timers (wasi:clocks)

  • 0.2: subscribe-duration(duration) -> pollable. You need to create a pollable and then poll it.
  • 0.3: sleep(duration) -> future<void>. You can directly await this sleep operation.

// Rust (wasi 0.2 style)
use wasi::clocks::monotonic_clock;
use wasi::io::poll;

fn main() {
    let duration = 1_000_000_000; // 1s
    // 1. Create a subscription (Resource)
    let pollable = monotonic_clock::subscribe_duration(duration);

    // 2. Block waiting for this pollable to be ready
    poll::poll_oneoff(&[&pollable]);

    // 3. Business logic
    println!("1 second passed!");
}

6. Conclusion

Although the WASI 0.3 update is labeled "Preview", it effectively completes the final piece of the WebAssembly Component Model puzzle.

  • For Standard Library Maintainers: 0.3 drastically simplifies API definitions, removing hundreds of lines of boilerplate code generated to simulate asynchrony.
  • For Application Developers: Finally, you can happily use async/await in Wasm and enjoy concurrency performance comparable to Go/Node.js.

If WASI 0.2 made WebAssembly components "Functional", then WASI 0.3 makes them "Fast" and "Joyful to Write".
